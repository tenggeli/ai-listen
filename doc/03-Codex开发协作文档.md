# listen Codex 开发协作文档

## 1. 文档目的
- 规范后续使用 Codex 进行 listen 项目开发时的上下文、目录、任务方式和交付标准。
- 降低反复解释需求的成本，让 Codex 可以持续稳定地推进项目。

## 2. Codex 开发总原则
- 先读文档再改代码。
- 优先保证 MVP 主链路。
- 每次任务尽量闭环：分析、编码、测试、输出结果。
- 不做无依据的假设，若需求存在多种高风险解释，先补充说明再推进。

## 7. Codex 执行规则

### 7.1 每次开发前
- 先扫描目录和已有文件。
- 判断是否存在用户未提交的修改，避免覆盖。
- 明确本次任务影响范围。

### 7.2 每次开发中
- 小步提交，保持结构清晰。
- 新增文件命名统一、目录统一。
- 核心逻辑必须写基础注释和必要测试。
- 优先复用已有组件、工具函数和模块。

### 7.3 每次开发后
- 说明改了什么。
- 说明哪些内容已验证，哪些未验证。
- 若存在风险或待补事项，要明确指出。

## 8. 代码规范建议

### 8.1 后端
- 控制器只做参数处理和响应。
- 业务逻辑放在 service。
- 数据访问放在 repository 或 dao。
- 错误码统一管理。
- 状态流转必须集中封装。

### 8.2 前端
- 页面与组件分层清晰。
- API 调用统一封装。
- 状态管理只放共享状态。
- 样式变量统一维护，避免散落硬编码。

### 8.3 数据库
- 表命名统一复数形式。
- 金额字段用整数。
- 必须考虑索引、唯一约束、状态字段。

## 9. 开发节奏建议

### 9.1 先后顺序
1. 基础工程
2. 用户/登录
3. 服务提供方入驻与审核
4. 服务项目
5. 订单主链路
6. 支付/结算/提现
7. 后台管理
8. 广场与声音
9. AI 与推荐
10. 风控与优化

### 9.2 每阶段产物
- 可运行代码
- 对应文档更新
- 测试结果
- 下一阶段任务建议

## 10. Codex 输出要求
- 涉及代码时，优先直接完成代码和文档，而不是只给建议。
- 涉及方案时，优先输出成文件，便于后续持续维护。
- 涉及 review 时，优先指出问题、风险、缺少测试点。

## 11. 关键约束
- 所有重要业务规则都要文档化。
- 所有状态机都要集中定义。
- 所有第三方配置都要可替换。
- 不直接把 AI 能力写死在页面逻辑里。
- 不在 MVP 阶段过早拆微服务。

- 未来每次进入新阶段时，应先补齐相关设计文档，再推进代码实现。
