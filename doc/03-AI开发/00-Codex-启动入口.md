# Listen Codex 启动入口

本文件是 Codex 进入 Listen 项目时的首读文档，用来快速统一目标、边界、优先级与执行顺序。

## 1. 当前目标

Listen 当前不是要一次性做完整平台，而是先建立适合持续 AI Coding 的最小开发骨架。

当前开发总目标：

- 先搭出清晰目录与分层架构
- 先实现用户 Web 的核心 AI 入口闭环
- 先实现后台最关键的运营管理能力
- 支付、结算、提现等复杂交易能力全部后置

## 2. 首次读取顺序

Codex 进入仓库后，建议固定按以下顺序建立上下文：

1. `doc/03-AI开发/00-Codex-启动入口.md`

## 3. 当前裁决规则

若文档之间出现冲突，统一按以下规则处理：

1. AI 主入口优先于传统功能堆叠
2. 用户 Web 优先于管理后台，管理后台优先于用户 App
3. 支付、结算、提现、VIP 等低优先能力不得阻塞 P0
4. 服务方当前优先按“平台内业务角色”实现，不急于独立成第四端

## 4. 当前推荐实现顺序

如果要让 Codex 连续开发，建议按下列顺序推进：

1. 用户 Web 首页 AI 主入口
2. 用户 Web AI 对话页
3. Go 后端 AI 会话与匹配次数接口
4. 用户 Web 服务页与推荐承接
5. 用户 Web 声音页
6. 管理后台服务方审核模块
7. 管理后台服务项目与声音内容管理
8. 订单骨架、评价、投诉

## 5. 当前最重要的业务边界

P0 必须聚焦：

- 用户登录与基础资料
- 首页 AI 主入口
- AI 对话页
- AI 推荐结果承接
- 服务浏览页
- 声音内容基础播放
- 我的页基础资产
- 后台审核、内容管理、服务管理、订单管理骨架

P0 可保留占位但不做深实现：

- 真支付
- 托管结算
- 提现
- VIP
- 广场复杂互动

## 6. 给 Codex 的编码约束

- 后端采用 Go + MySQL，按 `interface / application / domain / infrastructure` 分层。
- 前端采用 Vue，按 `views / components / domain / application / api` 分层。
- 业务逻辑不要直接写进 controller 或页面模板。
- 所有状态流转必须收敛到实体或领域服务。
- 每次实现只做一个清晰任务颗粒，不要一次铺开整个项目。

## 7. 常见误区

- 把首页做成普通信息流，而不是 AI 驱动入口
- 先实现支付，导致主链路迟迟不可运行
- 直接从 PRD 抽细节开写，没有先看 AI 开发文档
- 把服务方端直接做成独立第四端
- 在页面里散落业务枚举、状态判断和接口转换逻辑

## 8. 推荐任务写法

给 Codex 下任务时，优先使用这种格式：

```text
请基于 doc/03-AI开发/01-技术方案设计.md 与 doc/03-AI开发/02-AI-Coding-指导.md，
实现 [某个单一模块] 的最小可运行版本。

要求：
1. 明确端：用户 Web / 管理后台 / 用户 App / 后端
2. 明确范围：只做一个页面、一个模块或一个状态机
3. 明确是否允许 mock
4. 明确是否需要 migration、测试、API 契约
5. 默认不实现支付、结算、提现
```
