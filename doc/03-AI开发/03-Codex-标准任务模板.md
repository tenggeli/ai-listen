# Listen Codex 标准任务模板

本文件用于给 Codex 下达稳定、低歧义、可直接执行的开发任务。

原则：

- 一次只下达一个清晰任务
- 明确端、范围、约束、交付物
- 默认遵循 `doc/03-AI开发/00-Codex-启动入口.md`、`doc/03-AI开发/01-技术方案设计.md`、`doc/03-AI开发/02-AI-Coding-指导.md`
- 若仓库已有模块，优先要求“在现有代码基础上续写”
- 所有第三方交互默认先 mock，不要求真实对接
- 涉及 AI、短信、微信登录、支付时统一按 mock 结果实现
- 涉及数据时默认优先按数据库读取设计

```text

- 端：用户 Web / 管理后台 / 用户 App / Go 后端
- 页面或模块：【填写具体对象】
- 只做：【填写本次范围】
- 不做：【填写明确排除项】

实现要求：
1. 遵循现有分层与命名规范
2. 优先复用现有 ViewModel / UseCase / API Adapter / Entity
3. 业务逻辑不要直接写进页面模板或 controller
4. 如涉及第三方交互，统一先 mock，但要保留 adapter / gateway 接入位置
5. 状态要覆盖 idle / loading / success / empty / error
6. 代码尽量保持最小闭环，可运行、可扩展

8. 涉及数据尽量按 MySQL 读取设计
9. 若涉及 AI / 短信 / 微信登录 / 支付，统一先设计 mock 数据结构与 adapter 边界

1. 直接修改代码，不只给方案
2. 说明本次改了哪些文件
3. 说明如何验证

```

## 2. 页面开发模板

适用于：首页、AI 对话页、服务页、声音页、登录页、资料页、订单页、后台列表页。

- 对应原型 HTML

在现有代码基础上实现【页面名称】页面的最小可运行版本。

要求：
1. 使用现有技术栈与目录结构
2. 页面拆分为 views / components / domain / application / api
3. 页面必须优先复用已有 PageLoadState、ViewModel 或 API adapter
4. 页面必须包含：【列出核心区块】
5. 页面状态包含：idle、loading、success、empty、error
6. 若后端接口或第三方交互未完成，可先 mock，但要保留 adapter 层
7. 不要把复杂业务判断写在模板里
8. 若页面涉及列表 / 详情 / 资料 / 订单，数据结构要面向数据库读取设计
9. 若页面涉及 AI / 短信 / 微信登录 / 支付，统一先提供 mock 数据与 adapter 边界
10. 登录页默认手机号 / 验证码使用 13800113800 / 123
11. 输出后请说明页面入口、关键交互和验证方式

本次不做：
- 【如真实支付、复杂筛选、真实流式接口等】

## 3. 后端模块模板

适用于：AI 会话、登录资料、订单、服务方审核、声音内容等后端模块。

```text
请基于以下文档：

- doc/03-AI开发/00-Codex-启动入口.md
- doc/03-AI开发/01-技术方案设计.md
- doc/03-AI开发/02-AI-Coding-指导.md

在现有后端代码基础上实现 Go 后端【模块名称】的最小闭环。

要求：
1. 按 interface / application / domain / infrastructure 分层
2. 先定义或扩展 entity、repository interface、use case、dto、controller
3. 如有状态流转，必须集中到实体方法或领域服务
4. 提供最小可用接口：【列出接口】
5. 如涉及持久化变更，需要 migration
6. 需要基础单测或补充已有单测
7. 配置优先读取 ~/conf/listenbase.cof
8. 涉及业务数据尽量按 MySQL 读取设计
9. 若涉及第三方交互，先用 mock gateway 或 mock service
10. 若涉及支付，先提供 mock success 接口，不做真实支付

本次不做：
- 【如真实支付、真实 AI 网关、复杂风控等】
```

## 6. 重构模板

适用于：目录整理、分层拆分、DTO 收敛、状态机收口。

```text
请对【模块名称】进行小范围重构。

目标：
- 提升可维护性
- 符合当前 AI Coding 分层规范

要求：
1. 不改变现有对外行为
2. 只重构【指定范围】
3. 将散落逻辑收敛到 domain / application / api adapter
4. 优先复用已有对象与状态模型
5. 不要扩大成全项目重构
6. 若涉及配置或 mock 边界，也一并收敛
7. 输出时说明重构前问题、重构后结构、验证结果
```

## 7. 推荐填写方式

为了让 Codex 更稳定，建议每次任务至少写清楚这 9 个字段：

1. 做什么模块
2. 属于哪个端
3. 本次只做什么
4. 本次明确不做什么
5. 是否涉及第三方交互 mock
6. 是否需要测试 / migration / 接口
7. 是否要求复用现有模块
8. 是否涉及数据库读取设计
9. 是否涉及 `~/conf/listenbase.cof` 配置修改
