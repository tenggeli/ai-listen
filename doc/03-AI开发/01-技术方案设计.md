# Listen 面向 AI 开发的技术方案设计

## 0. 文档目标

本文件不是面向汇报的传统技术方案，而是面向 Codex / AI Coding 的开发输入文档。

目标：

- 用最少歧义定义产品范围、系统边界、对象模型、模块拆分、状态机与开发优先级
- 让 AI 可以按文档直接生成目录、接口、数据表、对象、页面、测试
- 基于当前代码现状与最新原型继续演进，而不是脱离仓库重写一套理想结构

约束：

- 开发范式：面向对象
- 技术栈：MySQL + Vue + Go + Nginx
- 产品形态：`用户 App`、`用户 Web`、`网页管理后台`
- 端优先级：`用户 Web`、`管理后台 Web`、`用户 App`
- 功能优先级：`AI 驱动主入口` 最高，`真实支付/结算` 最低
- 配置优先级：后端配置统一优先读取 `~/conf/listenbase.cof`

- 登录 / 注册占位链路
- 基础资料补全
- 性格设置（兴趣、MBTI、跳过）
- 首页 AI 主入口
- AI 对话页
- 服务浏览与服务方详情
- 声音页
- 我的页
- 订单创建与订单详情
- 评价与投诉骨架
- 后台服务方审核

### 2.2 当前允许 mock 的范围

允许先 mock：

- AI 推荐结果
- AI 对话自动回复
- AI 推荐话术、情绪标签、推荐理由
- 最终支付成功结果

但必须满足：

- mock 数据进入 `api adapter / service adapter` 层
- 页面、ViewModel、UseCase 不直接耦合假数据结构
- 后续可平滑替换为真实 API / AI 网关

### 2.3 当前必须面向数据库设计的范围

以下能力应尽量按数据库读取设计：

- 用户资料
- 性格标签与 MBTI

- 订单、订单状态、订单日志

- 后台运营配置

## 3. 系统总览

### 3.1 产品端划分

#### A. 用户 Web

定位：

- MVP 主战场
- 承接新用户引导、AI 入口、服务浏览、声音消费、订单与个人中心

#### B. 管理后台 Web

定位：

- 平台运营中枢

- 后续扩展服务项目、声音内容、订单、投诉、平台配置

#### C. 用户 App

定位：

- 暂为工程骨架
- 后续复用用户 Web 的领域模型与 API 契约

### 3.2 物理目录

当前实际目录：

- `web/apps/user-web`
- `web/apps/admin-web`
- `app/user-app`
- `backend`
- `backend/migrations`
- `doc`

### 3.3 逻辑分层

后端统一采用 4 层：

- `interface`：HTTP Controller、DTO、路由
- `application`：UseCase / AppService，编排业务流程
- `domain`：Entity、Repository Interface、DomainService
- `infrastructure`：MySQL Repository、Memory Repository、配置、第三方网关

前端统一采用 5 层：

- `views`：页面
- `components`：纯 UI 组件
- `application`：页面级用例编排、状态协调
- `domain`：TS 实体、枚举、查询对象
- `api`：API Adapter，承接 Mock 与 HTTP 实现

## 4. 配置设计

### 4.1 配置源

后端配置统一优先读取：

- `~/conf/listenbase.cof`

建议配置分类：

- `server`
- `mysql`
- `repository`
- `ai`
- `mock`
- `admin`

建议最少支持的配置项：

- `server.port`
- `mysql.dsn`
- `repository.driver` (`memory/mysql`)
- `ai.mode` (`mock/real`)
- `mock.enable_payment_success`

### 4.2 配置读取策略

- 服务启动时统一装载配置对象
- Application / Infrastructure 通过配置对象获取依赖参数
- 不允许在 UseCase 中直接读取环境变量
- 不允许把数据库 DSN、AI 开关硬编码进业务代码

## 5. 面向对象建模原则

### 5.1 后端 OOP 规则

- 所有核心业务都要先定义领域对象，再写控制器
- Controller 不直接写 SQL
- UseCase 不直接依赖 HTTP 框架
- Repository 只负责持久化，不承载复杂业务判断
- 状态流转必须收口到 Entity 方法或 DomainService

当前已落地示例：

- `Session.AppendMessage()` 负责消息追加与会话时间更新
- `DailyQuota.Consume()` 负责每日次数扣减
- `Provider.Approve()` / `Reject()` / `RequireSupplement()` 负责审核状态流转

### 5.2 前端 OOP 规则

- 页面只负责展示和事件分发
- 业务规则写入 `domain` 或 `application`
- API 返回数据先转为前端实体对象，再进入页面
- 枚举、状态映射、权限判断不可散落在组件模板里

## 6. 当前核心领域模型

### 6.1 领域上下文

按当前实现与最新原型，建议拆为 8 个上下文：

1. `ai_companion`：AI 会话、消息、匹配次数、推荐结果
2. `provider`：服务方资料、审核状态流转
3. `identity`：登录、用户身份、基础资料、性格设置
4. `service_discovery`：服务分类、服务方、服务项目、详情页承接
5. `audio_content`：声音分类、声音内容、播放记录
6. `order`：订单、订单状态、订单日志、订单详情
7. `feedback`：评价、投诉、售后反馈
8. `payment`：支付确认、支付结果占位、后续真实支付网关

## 8. 数据库设计原则

### 8.1 总原则

- 详情、列表、字典、配置尽量设计为从数据库读取
- mock 只用于 AI 输出与支付结果占位，不替代核心业务持久化
- 新增后端模块时优先补 migration，再补 repository / usecase / controller

## 10. 落地策略总结

当前 Listen 的推荐落地策略是：

- `数据库优先`：能进数据库的能力尽量进数据库
- `AI 先 mock`：AI 输出先 mock，再替换真实模型
- `支付最后接`：支付流程先 mock 成功，不阻塞订单与售后链路
- `配置外置`：后端配置优先从 `~/conf/listenbase.cof` 读取
- `按原型推进`：开发顺序以 `doc/04-原型素材` 最新原型流为主线
