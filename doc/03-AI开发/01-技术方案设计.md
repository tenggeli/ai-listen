# Listen 面向 AI 开发的技术方案设计

## 0. 文档目标

本文件不是面向汇报的传统技术方案，而是面向 Codex / AI Coding 的开发输入文档。

目标：

- 用最少歧义定义产品范围、系统边界、对象模型、模块拆分、状态机与开发优先级
- 让 AI 可以按文档直接生成目录、接口、数据表、对象、页面、测试
- 基于当前代码现状继续演进，而不是脱离仓库重写一套理想结构

约束：

- 开发范式：面向对象
- 技术栈：MySQL + Vue + Go + Nginx
- 产品形态：`用户 App`、`用户 Web`、`网页管理后台`
- 端优先级：`用户 Web`、`管理后台 Web`、`用户 App`
- 功能优先级：`AI 驱动主入口` 最高，`支付/订单结算` 最低

## 3.3 逻辑分层

后端统一采用 4 层：

- `interface`：HTTP Controller、DTO、路由
- `application`：UseCase / AppService，编排业务流程
- `domain`：Entity、Repository Interface、DomainService
- `infrastructure`：MySQL Repository、Memory Repository、配置、第三方网关

前端统一采用 5 层：

- `views`：页面
- `components`：纯 UI 组件
- `application`：页面级用例编排、状态协调
- `domain`：TS 实体、枚举、查询对象
- `api`：API Adapter，承接 Mock 与 HTTP 实现

## 4. 面向对象建模原则

## 4.1 后端 OOP 规则

- 所有核心业务都要先定义领域对象，再写控制器
- Controller 不直接写 SQL
- UseCase 不直接依赖 HTTP 框架
- Repository 只负责持久化，不承载复杂业务判断
- 状态流转必须收口到 Entity 方法或 DomainService

当前已落地示例：

- `Session.AppendMessage()` 负责消息追加与会话时间更新
- `DailyQuota.Consume()` 负责每日次数扣减
- `Provider.Approve()` / `Reject()` / `RequireSupplement()` 负责审核状态流转

## 4.2 前端 OOP 规则

- 页面只负责展示和事件分发
- 业务规则写入 `domain` 或 `application`
- API 返回数据先转为前端实体对象，再进入页面
- 枚举、状态映射、权限判断不可散落在组件模板里

当前已落地示例：

- `HomePageViewModel`
- `ChatPageViewModel`
- `ProviderReviewViewModel`
- `AiHomeAggregate`
- `AiSession`

## 5. 当前核心领域模型

## 5.1 领域上下文

按当前实现优先拆为 3 个已落地上下文与 5 个待扩展上下文：

已落地：

1. `ai_companion`：AI 会话、消息、匹配次数、推荐结果
2. `provider`：服务方资料、审核状态流转
3. `operation_admin`：后台审核操作入口

待扩展：

4. `identity`：身份、登录、用户资料
5. `service_discovery`：服务浏览、筛选、推荐结果承接
6. `audio_content`：声音内容、分类、播放记录
7. `order`：订单、评价、投诉、状态流转
8. `payment`：支付、结算、提现

## 6. 当前接口设计

## 6.1 用户侧 AI 接口

- `GET /api/v1/ai/match/remaining?user_id=...`
- `POST /api/v1/ai/match`
- `POST /api/v1/ai/sessions`
- `GET /api/v1/ai/sessions/{id}`
- `POST /api/v1/ai/sessions/{id}/messages`

说明：

- 当前匹配候选由 `MockMatchService` 生成
- 每日匹配上限为 `5`
- 当前消息追加仅保存用户消息，不生成 AI 回复

## 6.2 管理后台接口

- `GET /api/v1/admin/providers`
- `GET /api/v1/admin/providers/{id}`
- `POST /api/v1/admin/providers/{id}/approve`
- `POST /api/v1/admin/providers/{id}/reject`
- `POST /api/v1/admin/providers/{id}/require-supplement`

## 7. 数据存储设计

## 7.1 已落地表

- `ai_sessions`
- `ai_messages`
- `ai_match_requests`
- `ai_match_results`
- `ai_daily_quotas`
- `providers`
- `provider_profiles`
- `provider_review_records`

## 7.2 当前存储策略

- 默认使用 `memory`
- 设置 `LISTEN_REPOSITORY_DRIVER=mysql` 后切换到 MySQL
- MySQL 初始化失败时自动回退到 `memory`

## 8. 当前技术约束与缺口

技术约束：

- 后端使用标准库 `net/http`
- 前端使用 Vue 3 + Vue Router + Vite
- 当前未接入统一鉴权
- 当前未接入真实 AI 大模型网关

主要缺口：

- AI 对话页缺少 AI 自动回复能力
- AI 匹配结果尚未持久化写入 `ai_match_requests` / `ai_match_results`
- 用户 Web 仅完成首页和对话页
- 管理后台仅完成服务方审核模块
- 用户 App 仍为占位骨架
