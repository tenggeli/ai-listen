# Listen 面向 AI 开发的技术方案设计（按 2026-04-02 仓库代码校准）

## 0. 文档目标

本文件是给 Codex / AI Coding 的执行输入，不是汇报材料。

目标：

- 用最少歧义描述当前真实系统状态
- 让 AI 可直接按文档继续扩展代码
- 统一“已完成/未完成”口径，避免重复开发

约束：

- 技术栈：MySQL + Vue + Go + Nginx
- 产品形态：用户 Web、服务方管理后台、平台管理后台、用户 App、Go 统一接口服务
- 端优先级：用户 Web > 平台管理后台 > 服务方管理后台 > 用户 App
- 功能优先级：AI 主入口与核心链路优先，第三方真实对接最后
- 配置优先级：后端配置优先读取 `~/conf/listenbase.cof`

### 1.5 Go 后端

后端统一按“一个接口服务”设计，不拆独立微服务。

接口分组口径：

- 用户侧接口：`/api/v1/...`
- 平台管理侧接口：`/api/v1/admin/...`
- 服务方侧接口：`/api/v1/provider/...`
- 基础设施接口：`/healthz`

新增接口命名规范：

- 所有路径使用小写英文
- 路径 segment 使用短横线，不使用驼峰、下划线、拼音
- 资源名优先使用复数名词
- 查询列表：`GET /resources`
- 查询详情：`GET /resources/{id}`
- 创建资源：`POST /resources`
- 更新资源：`PUT /resources/{id}`
- 动作型接口：`POST /resources/{id}/action`
- 单例资源允许使用单数名词，例如 `profile`

推荐动作命名：

- 审核类：`approve`、`reject`、`require-supplement`
- 订单履约类：`accept`、`depart`、`arrive`、`complete`
- 支付模拟类：`pay/mock-success`

不推荐写法：

- `/api/v1/getOrders`
- `/api/v1/providerOrderList`
- `/api/v1/admin/provider/reviewPass`

历史接口说明：

- 以下规则主要约束新增接口，不回溯要求当前已存在接口全部改名
- 当前历史接口例外包括：

## 2. 产品边界（当前阶段）

### 2.1 P0/P1 必做范围

- 保持用户引导链路稳定（登录、资料、性格）
- 服务浏览与服务方详情稳态
- 声音页 HTTP 接口稳态
- 我的页与设置入口
- 订单创建、支付确认（mock-success）、订单列表/详情稳态
- 评价/投诉最小闭环稳态
- 设置页偏好能力后端持久化稳态（MySQL），并保留 memory 本地兜底
- 平台管理后台鉴权骨架与运营模块扩展
- 服务方管理后台前端工程与核心工作台

### 2.2 第三方交互策略

当前统一 mock，不做真实对接：

- AI 网关
- 短信
- 微信授权
- 支付

要求：

- 所有 mock 收口到 adapter/gateway 层
- 页面/ViewModel/UseCase 不直接耦合第三方 SDK 结构

### 2.3 必须面向数据库设计的范围

- order
- feedback
- payment_record
- admin / provider_console 配置类数据

说明：

- `identity` 已完成 MySQL 落地，不再是待补状态
- `service_discovery` 已有 memory/mysql 双实现
- `order`、`feedback` 已完成后端最小闭环与 MySQL 落地

## 7. 当前推荐开发策略

1. 不重做已完成的服务浏览与声音接口
2. 在已落地设置持久化与后台鉴权骨架上继续增量扩展
3. 订单链路在现有最小闭环基础上扩展状态机与运营能力
4. 平台管理后台继续在已落地审核模块上增量扩展
5. 服务方管理后台独立起工程，优先做工作台与履约链路
6. 用户 App 继续后置
