# Listen 面向 AI 开发的技术方案设计（按最新代码校准）

## 0. 文档目标

本文件是给 Codex / AI Coding 的执行输入，不是汇报材料。

目标：

- 用最少歧义描述当前真实系统状态
- 让 AI 可直接按文档继续扩展代码
- 统一“已完成/未完成”口径，避免重复开发

约束：

- 技术栈：MySQL + Vue + Go + Nginx
- 产品形态：用户 Web、管理后台 Web、用户 App
- 端优先级：用户 Web > 管理后台 Web > 用户 App
- 功能优先级：AI 主入口与核心链路优先，第三方真实对接最后
- 配置优先级：后端配置优先读取 `~/conf/listenbase.cof`

## 1. 当前实现现状

### 1.1 用户 Web

已实现页面与路由：

- `/auth` 登录页（短信 mock 登录 + 微信 mock 登录）
- `/profile/setup` 基础资料补全
- `/personality/setup` 性格设置（兴趣、MBTI、跳过）
- `/home` AI 主入口
- `/chat` AI 对话页
- `/sound` 声音页

已实现能力：

- 路由守卫：未登录跳转 `/auth`，登录后按资料/性格完成度引导
- 登录默认值：`13800113800 / 123`
- 身份会话本地保存：`AuthSession`
- `AiApi` HTTP 实现（首页、匹配、会话）
- `SoundApi` Mock + HTTP 双实现（HTTP 端点当前后端未提供）

未实现页面：

## 2. 产品边界（当前阶段）

### 2.1 P0/P1 必做范围

- 保持用户引导链路稳定（登录、资料、性格）
- 服务浏览与服务方详情
- 声音后端接口补齐并对齐前端 HTTP
- 我的页
- 订单创建、支付确认（mock-success）、订单详情
- 评价/投诉最小闭环
- 管理后台鉴权骨架与运营模块扩展

### 2.2 第三方交互策略

当前统一 mock，不做真实对接：

- AI 网关
- 短信
- 微信授权
- 支付

要求：

- 所有 mock 收口到 adapter/gateway 层
- 页面/ViewModel/UseCase 不直接耦合第三方 SDK 结构

### 2.3 必须面向数据库设计的范围

- identity（当前已建领域模型，需补 mysql 落地）
- service_discovery
- audio_content
- order
- feedback
- payment_record
- admin 配置类数据
