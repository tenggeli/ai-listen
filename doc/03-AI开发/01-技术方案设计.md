# Listen 面向 AI 开发的技术方案设计

## 0. 文档目标

本文件不是面向汇报的传统技术方案，而是面向 Codex / AI Coding 的开发输入文档。

目标：

- 用最少歧义定义产品范围、系统边界、对象模型、模块拆分、状态机与开发优先级
- 让 AI 可以按文档直接生成目录、接口、数据表、对象、页面、测试
- 优先保障 P0 可落地，其次再扩展复杂交易和支付

约束：

- 开发范式：面向对象
- 技术栈：MySQL + Vue + Go + Nginx
- 产品形态：`用户 App`、`用户 Web`、`网页管理后台`
- 端优先级：`用户 Web`、`管理后台 Web`、`用户 App`
- 功能优先级：`AI 驱动主入口` 最高，`支付/订单结算` 最低

推断说明：

- 产品文档存在“服务方端”完整 PRD，但本轮最终产品形态只要求 3 端，因此将“服务方能力”先视为平台内业务角色，不单独形成第四端产品
- P0/P1 中服务方相关能力，优先由后台审核、配置与订单状态流转支持；服务方独立工作台可在后续作为 `用户 App` 的角色模式或单独端继续扩展

## 1. 产品边界

## 1.1 P0 必做范围

P0 只做能证明 Listen 核心价值的最小闭环：

- 用户登录与基础资料
- 首页 AI 主入口
- AI 对话页
- AI 推荐结果承接
- 服务浏览页
- 声音内容页基础播放
- 我的页基础资产
- 后台：服务方审核、服务项目管理、内容管理、订单管理、投诉处理、运营配置

P0 不追求完整交易闭环，支付仅保留占位接口与状态扩展位，不作为上线阻塞项。

## 1.2 P1 范围

- 订单创建
- 订单状态机
- 评价与投诉
- 服务方资料审核流
- 手动筛选与 AI 推荐联动
- 基础通知中心

## 1.3 P2 范围

- 实际支付
- 托管结算
- 提现
- VIP
- 广场动态
- 推荐系统增强

## 2. 系统总览

## 2.1 产品端划分

### A. 用户 Web

定位：

- MVP 主战场
- 承载首页 AI 入口、AI 对话、服务浏览、声音、我的

### B. 管理后台 Web

定位：

- 平台运营中枢
- 承载审核、内容管理、服务管理、订单管理、投诉处理、运营配置

### C. 用户 App

定位：

- 复用用户 Web 的业务能力
- 强化语音输入、推送、状态通知、移动端体验

## 2.2 后端系统拆分

单体优先，模块化单体实现，不做微服务。

原因：

- 当前需求仍在快速变化
- AI Coding 更适合在单仓内做一致性演进
- 领域复杂度不足以支撑微服务拆分收益

建议物理结构：

- `web/user-web`：用户 Web
- `web/admin-web`：管理后台 Web
- `app/user-app`：用户 App
- `backend`：Go API 服务
- `deploy/nginx`：Nginx 配置
- `doc`：产品、技术、AI coding 文档

## 2.3 逻辑分层

后端统一采用 4 层：

- `interface`：HTTP Controller、DTO、路由
- `application`：UseCase / AppService，编排业务流程
- `domain`：Entity、ValueObject、DomainService、Repository Interface
- `infrastructure`：MySQL Repository、第三方 AI 网关、缓存、文件存储

前端统一采用 4 层：

- `pages/views`：页面
- `components`：纯 UI 组件
- `application`：页面级用例编排、状态协调
- `domain`：TS 实体、枚举、查询对象、客户端服务对象

## 3. 面向对象建模原则

## 3.1 后端 OOP 规则

- 所有核心业务都要先定义领域对象，再写控制器
- Controller 不直接写 SQL
- UseCase 不直接依赖 HTTP 框架
- Repository 只负责持久化，不承载复杂业务判断
- 状态流转必须收口到 Entity 方法或 DomainService

示例：

- `Order.Accept()` 负责校验是否允许接单
- `ProviderProfile.SubmitForReview()` 负责驱动审核状态切换
- `AiSession.AppendMessage()` 负责维护会话消息序列

## 3.2 前端 OOP 规则

- 页面只负责展示和事件分发
- 业务规则写入 `domain` 或 `application`
- API 返回数据先转为前端实体对象，再进入页面
- 枚举、状态映射、权限判断不可散落在组件模板里

示例：

- `AiConversationSession`
- `MatchCandidate`
- `ServiceProviderCard`
- `OrderDetailVM`

## 5. 状态机定义

## 5.1 AI 匹配状态

`idle -> listening -> recognizing -> clarifying -> matched | failed | limited`

规则：

- 单用户单日 AI 匹配上限 5 次
- 匹配结果固定返回 3 个候选
- 匹配失败不扣减次数，由 application 层决定是否回滚次数

## 5.2 服务方审核状态

`draft -> submitted -> under_review -> approved | rejected | supplement_required | frozen`

## 5.3 服务项目审核状态

`draft -> pending_review -> approved | rejected | offline`

## 5.4 订单状态

P1 起启用完整状态机：

`pending_payment -> pending_accept -> accepted -> departed -> arrived -> service_confirming -> in_service -> pending_finish_confirm -> completed | cancelled | complaint_processing`

P0 中只要求保留状态结构，不强制打通全流程。

## 5.5 投诉状态

`created -> under_review -> resolved_refund | resolved_partial_refund | resolved_release_fund | rejected`

## 8. AI 能力技术设计

## 8.1 AI 子系统职责

AI 子系统只负责：

- 情绪识别
- 意图抽取
- 追问判断
- 结果推荐说明文案
- 页面跳转建议

AI 子系统不直接负责：

- 最终排序落库逻辑
- 订单创建
- 权限判断
- 支付执行

## 8.2 AI 调用分层

- `AiGateway`：统一调用外部大模型
- `PromptBuilder`：按场景构造提示词
- `IntentParser`：解析模型输出为结构化对象
- `MatchAssembler`：结合数据库候选生成最终 3 个推荐对象
- `SafetyGuard`：敏感内容判定与拦截

## 8.3 AI 结构化输出要求

所有 AI 输出必须先落成 JSON，再转换为 UI 文案。

标准结构：

```json
{
  "emotion_tags": ["lonely"],
  "scene_tags": ["dinner", "movie"],
  "need_clarify": true,
  "clarify_question": "你更想先看电影还是先吃饭？",
  "recommended_action": "match_provider",
  "reasoning_summary": "用户在表达即时线下陪伴需求"
}
```

## 8.4 AI 失败兜底

- 超时：展示固定温和话术并建议手动筛选
- 无结果：降级到热门服务方 + 声音推荐
- 高风险内容：转入安全提示，不继续推荐线下服务

## 12. 非功能要求

## 12.1 安全

- 登录态使用 JWT + Refresh Token 或服务端 Session 二选一，优先 JWT
- 敏感接口做角色权限校验
- 图片与文本内容接入审核链路
- 手机号、证件号、支付账号都要脱敏展示

## 12.2 性能

- 首页首屏接口小而快
- AI 对话接口采用流式输出
- 列表查询默认分页
- 后台筛选查询必须有索引

## 12.3 可观测性

- 所有请求生成 `request_id`
- 关键状态流转记录 `operation_logs`
- AI 请求与解析失败要有单独日志分类

## 14. AI 开发执行约束

- 每次只实现一个 bounded context 内的一组能力
- 先建对象、枚举、仓储接口，再写 use case
- 前后端都先写契约，再补页面和仓储
- 每个接口都必须有 DTO、校验、错误码、测试点
- 每个状态机都必须有状态迁移表，禁止直接字符串跳转

## 15. 本文档的直接开发结论

当前最合理的实施顺序不是“先订单支付”，而是：

1. 用户 Web 首页 AI 主入口
2. AI 对话与结构化意图识别
3. 服务推荐承接页
4. 声音页
5. 管理后台审核与内容配置
6. 我的页与基础资产
7. 订单状态骨架
8. 评价与投诉
9. 支付与结算

这一路径最符合产品定位，也最符合你给出的优先级要求。
