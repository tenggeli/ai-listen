# ai-listen 技术架构详细设计文档

## 1. 文档说明
本文件用于定义 ai-listen 项目的技术实现架构，指导后端开发、前端开发、运维部署、接口联调和后续扩展。

适用范围：
- 用户端 APP
- 服务提供方端 APP
- 平台管理后台
- 后端 API 服务
- 数据存储与部署体系

技术栈约束：
- 后端：Go
- 数据库：MySQL
- 前端：Vue
- 网关：Nginx

建议补充组件：
- Redis
- WebSocket
- 对象存储
- 消息队列
- 日志与监控系统

---

## 2. 架构目标

### 2.1 目标
- 支撑 APP 端高频访问和订单实时流转
- 支撑 AI 匹配、订单履约、支付结算、评价、内容等核心能力
- 前期开发效率高，后期可平滑扩展
- 保障订单状态一致性和支付安全性
- 为广场、声音、VIP、推荐优化等模块预留扩展能力

### 2.2 设计原则
- **先模块化单体，后按需拆分**
- **核心交易链路优先稳定**
- **读写分层，业务边界清晰**
- **状态变更必须可追踪**
- **外部依赖必须可替换**
- **敏感数据必须脱敏/加密**

---

## 3. 总体技术架构

## 3.1 架构分层
整体建议采用五层架构：

1. **客户端层**
   - 用户端 APP
   - 服务提供方端 APP
   - 管理后台 Web

2. **接入层**
   - Nginx
   - HTTPS
   - API 路由转发
   - 静态资源分发
   - WebSocket 转发

3. **应用层**
   - Go API 服务
   - 后台管理服务
   - Worker 异步任务

4. **能力层**
   - 认证能力
   - AI 匹配能力
   - 支付能力
   - 消息推送能力
   - 文件上传能力
   - 实名认证能力

5. **数据层**
   - MySQL
   - Redis
   - 对象存储
   - 日志/监控

---

## 3.2 架构形态建议
### 阶段一：模块化单体
适合 MVP 和前期快速迭代。

特点：
- 一个主 API 服务
- 一个后台服务（可与主服务同进程或独立部署）
- 一个异步 Worker
- 一个 MySQL
- 一个 Redis

### 阶段二：按域拆分服务
当订单量、音频内容、推荐量上来后，再按业务域拆分：
- 用户服务
- 服务提供方服务
- 订单服务
- 结算服务
- 内容服务
- 音频服务
- AI 推荐服务
- 消息服务

## 4. 前端架构设计

## 4.1 用户端 / 服务端 APP
建议采用：
- Vue3
- uni-app（便于 iOS/Android 跨端）
- Pinia
- Axios
- WebSocket

### 工程划分建议
#### 方案 A：单 APP 多身份切换
- 用户与服务提供方共用一个 APP
- 登录后根据身份展示不同入口
- 优点：维护成本低
- 缺点：逻辑会更复杂

#### 方案 B：双 APP
- 用户端独立 APP
- 服务提供方端独立 APP
- 优点：定位清晰、流程更纯粹
- 缺点：开发与维护成本更高

前期建议：
- **MVP 优先采用单 APP + 身份切换模式**

## 4.2 管理后台
建议采用：
- Vue3
- Vite
- Element Plus
- Pinia
- Axios

后台模块：
- 仪表盘
- 服务方审核
- 订单中心
- 用户管理
- 服务方管理
- 财务中心
- 内容审核
- 举报投诉处理

## 5. 后端架构设计

## 5.1 后端框架建议
建议使用：
- Go 1.22+
- Gin 或 Fiber（二选一）
- GORM 或 sqlx（二选一）
- Zap 日志
- Viper 配置管理
- Redis 客户端

建议优先：
- **Gin + GORM + Zap + Redis**

## 5.2 项目目录建议
```text
ai-listen/
  cmd/
    api/
    admin/
    worker/
  internal/
    app/
    config/
    router/
    middleware/
    handler/
    service/
    repository/
    model/
    dto/
    pkg/
  scripts/
  deploy/
  docs/
```

## 5.3 分层职责
### handler 层
- 接收请求
- 参数校验
- 调用 service
- 返回统一结构

### service 层
- 核心业务逻辑

- 状态流转控制
- 外部依赖编排

### repository 层
- 数据访问
- 查询封装

### model 层
- 数据模型定义

### dto 层
- 请求和响应结构

## 7. 订单状态机设计

订单状态建议如下：
- 1：待支付
- 2：待接单
- 3：已接单
- 4：已出发
- 5：已到达
- 6：服务中
- 7：待确认完成
- 8：已完成
- 9：已取消
- 10：售后处理中
- 11：已退款
- 12：已结算

### 状态流转规则
- 待支付 -> 待接单
- 待接单 -> 已接单
- 已接单 -> 已出发
- 已出发 -> 已到达
- 已到达 -> 服务中
- 服务中 -> 待确认完成
- 待确认完成 -> 已完成
- 已完成 -> 已结算

所有状态流转必须：
- 校验当前状态是否合法
- 写入订单轨迹表
- 记录操作者
- 关键步骤加事务

## 8. 数据存储设计

## 8.1 MySQL
用途：
- 用户资料
- 服务方资料
- 服务项目
- 订单
- 支付
- 提现
- 评价
- 广场内容
- 音频内容

建议：
- InnoDB
- utf8mb4
- datetime 存 UTC
- 核心表添加逻辑删除字段

## 8.2 Redis
用途：
- Token 缓存
- 验证码
- 首页推荐临时缓存
- 热门榜单
- 订单短期状态缓存
- 幂等控制
- 分布式锁

### 典型 Key 示例
- `login:sms:{mobile}`
- `token:user:{uid}`
- `order:lock:{orderId}`
- `home:match:{uid}`
- `rank:provider:{cityId}`

## 8.3 对象存储
用途：
- 用户头像
- 服务方照片
- 动态图片
- 评价图片
- 音频文件
- 实名资料附件

建议接入：
- 阿里云 OSS / 腾讯云 COS / MinIO

## 9. AI 推荐架构设计

## 9.1 前期方案
MVP 不建议一上来做复杂大模型决策系统，可采用：
- 规则引擎 + 标签匹配 + 权重排序
- 语音识别由第三方完成
- 文本意图识别优先规则化

## 9.2 推荐处理流程
1. 接收用户输入
2. 解析场景、时间、预算、地点
3. 查询候选服务方
4. 基于权重排序
5. 生成 3 个推荐对象
6. 返回匹配理由

## 9.3 推荐排序维度
- 场景匹配度
- 地理距离
- 可服务时间
- 价格匹配度
- 历史评分
- 完单率
- 投诉率
- 用户偏好权重
- VIP/运营加权

## 11. 支付与结算架构

## 11.1 支付渠道
建议优先接入：
- 微信支付
- 支付宝（可后续）

## 11.2 支付流程
1. 创建订单
2. 调用支付下单
3. 前端拉起支付
4. 支付回调
5. 订单状态更新
6. 记录支付流水

## 11.3 结算流程
1. 服务完成
2. 用户确认或超时确认
3. 判断是否有投诉
4. 计算平台佣金
5. 计入服务方余额
6. 服务方申请提现

## 11.4 风险控制
- 支付回调幂等
- 金额校验
- 重复回调防重
- 异议订单冻结结算

## 15. 非功能设计

## 15.1 性能
- 首页首屏可用时间 < 2 秒
- 核心接口平均响应 < 500ms
- 推荐接口优先缓存热点数据

## 15.2 稳定性
- 核心交易接口要支持重试
- 状态流转接口需幂等
- 异步任务需补偿机制

## 15.3 可维护性
- 模块边界清晰
- 统一错误码
- 统一日志规范
- 统一配置规范

## 17. 结论
ai-listen 的技术架构应优先围绕 **移动 APP 的稳定交易闭环** 来设计，而不是一开始追求复杂的微服务化。

最关键的技术点有三个：
1. **首页 AI 匹配体验要顺**
2. **订单状态机要稳**
3. **支付结算与风控要严**

前期建议：**模块化单体 + Redis + MySQL + 对象存储 + WebSocket**，先把 MVP 跑通，再逐步拆分与升级。
