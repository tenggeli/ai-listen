# ai-listen APP 项目设计文档

## 1. 项目概述

### 1.1 项目名称
ai-listen（社交陪伴平台）

### 1.2 项目定位
ai-listen 最终交付形态为 **移动端 APP**，定位为一个以 **AI 智能匹配 + 线下陪伴服务 + 声音内容陪伴 + 社交广场互动** 为核心的社交陪伴平台。

平台连接两类核心角色：
- **用户方**：购买线下陪伴、聊天、饭搭子、影搭子、陪玩、心理疏导等服务
- **服务提供方**：通过实名认证、资料审核、服务配置、接单履约，为用户提供陪伴服务并获取收益

### 1.3 核心目标
本项目强调以下产品理念：
- **把 AI 拟人化**：用户面对的是一个懂自己的智能伙伴，而不是工具菜单
- **把操作指令化**：一句话完成筛选、匹配、邀约、下单
- **把信息可视化且极简化**：保留未来感与科技感，减少传统列表型压迫感
- **把线下陪伴产品化、标准化、安全化**：提升履约效率、交易确定性与平台安全能力

### 1.4 产品形态说明
本项目建议采用以下产品形态：
- **用户端 APP**：iOS / Android
- **服务提供方 APP**：可独立端，也可通过同一 APP 的身份切换模式实现
- **管理后台 Web**：平台审核、订单、财务、运营、风控统一后台

### 1.5 技术选型
- **后端**：Go
- **数据库**：MySQL
- **前端**：Vue（建议基于 uni-app / Vue3 体系实现跨端 APP）
- **网关/部署**：Nginx

建议补充：
- Redis
- 对象存储（OSS / COS / MinIO）
- WebSocket
- 消息队列
- 第三方实名认证、支付、语音识别与推送服务

## 2. 产品整体架构

### 2.1 APP 底部五大板块
- **首页**
- **服务**
- **广场**
- **声音**
- **我的**

### 2.2 三端协同体系
1. **用户端（C 端 APP）**
   - AI 输入需求
   - 匹配服务方
   - 浏览服务方资料
   - 下单、支付、投诉、评价、打赏、收藏、VIP

2. **服务提供方端（S 端 APP）**
   - 注册认证
   - 管理服务项目与价格
   - 接单、出发、到达、开始、完单
   - 收益提现
   - 评价用户、发布动态

3. **平台管理端（B 端 Web）**
   - 审核服务提供方
   - 管理订单与争议
   - 财务结算
   - 风控与报表
   - 内容审核与运营

### 2.3 APP 核心原则
- 单手可操作
- 首页操作路径最短
- 重要状态可前台提醒、通知栏提醒、锁屏提醒
- 视觉极简但层级清晰
- 以语音输入和卡片化交互为核心

## 3. 核心功能模块设计

## 3.1 首页（用户端核心入口）
### 3.1.1 功能定位
首页是 APP 最重要的下单入口，目标是让用户用最短路径表达需求并获得匹配结果。

### 3.1.2 页面特征
- 深色科技感背景
- 磨砂玻璃质感浮层
- 中部偏下核心呼吸球
- 上方 AI 推荐环绕内容
- 无传统九宫格入口

### 3.1.3 核心能力
- 语音输入
- 文字输入
- AI 推荐 3 位候选服务方
- 手动高级筛选
- 匹配解释
- 一键发起邀约

### 3.1.4 APP 化增强能力
- 支持麦克风权限授权与引导
- 支持语音转文字回显
- 支持震动反馈、粒子动画、卡片翻转动效
- 支持系统推送“已为你找到 3 位合适对象”
- 支持锁屏/前台状态提醒

## 3.2 服务（服务提供方工作台）
### 3.2.1 功能定位
服务板块为服务提供方的移动工作台，强调高频、实时、可移动、可通知。

### 3.2.2 功能清单

- 定向邀约
- 订单详情
- 出发 / 到达 / 开始 / 完单
- 地图导航跳转
- 收益、评分、等级
- 历史订单
- 异常取消

### 3.2.3 APP 化增强能力
- 新订单推送
- 出发/到达快捷操作
- 到达时定位校验
- 服务中计时浮层
- 后台运行时状态保活提示

## 5. APP 业务流程设计

### 5.1 用户下单流程
1. 打开首页
2. 语音/文字输入需求
3. AI 解析并推荐 3 位候选人
4. 查看卡片并筛选
5. 发起邀约
6. 支付到平台
7. 服务方接单
8. 服务方出发并到达
9. 现场确认开始计时
10. 服务完成
11. 用户确认或超时自动确认
12. 平台结算
13. 双方评价

### 5.2 服务提供方履约流程
1. 在线并切换为空闲
2. 接收订单/主动邀约
3. 确认接单
4. 出发
5. 到达后确认
6. 开始服务
7. 提交完单
8. 等待结算
9. 提现

### 5.3 异常流程
- 用户爽约
- 服务方爽约
- 到达无法联系
- 服务争议
- 投诉与冻结结算

## 10. 结论
ai-listen 应以 **APP 为最终核心载体**，围绕“AI 指令式交互 + 线下陪伴标准化履约 + 内容化信任建设 + 声音情绪陪伴”形成产品闭环。

项目建议建设顺序：
1. 先做交易闭环
2. 再做内容与声音
3. 最后做智能深化与城市化运营
