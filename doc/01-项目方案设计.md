# listen 项目方案设计（精简版）

## 1. 项目定位
listen 是一个以 AI 交互为入口的陪伴服务平台，围绕四条主线建设：
- 用户消费：浏览内容、下单、支付、履约确认
- 服务供给：服务方入驻、配置、接单、履约
- 内容互动：广场、音频、评论、点赞
- 平台治理：审核、风控、财务、RBAC

## 2. 当前工程落位（与代码一致）
- 后端：`backend/`（Go + Gin，统一 API）
- 用户 Web：`web/apps/user-web`
- 管理 Web：`web/apps/admin-web`
- 用户 App：`app/apps/user-app`
- 管理 App：`app/apps/admin-app`
- App 共享层：`app/packages/shared`
- 历史目录：`app/listen-app`（仅保留迁移说明）

## 3. 当前已实现能力（按接口层）
- 认证：短信登录、微信登录占位、token 刷新
- 用户侧：`/users/me`、个人资料、我的订单/收藏
- 服务方侧：入驻申请、资料更新、服务项目、工作状态、履约动作
- 交易：创建订单、状态流转、支付创建/查询、支付回调验签与幂等基础骨架
- 内容：帖子、评论、点赞、评价、投诉已具备基础创建/查询链路；音频、AI 匹配仍偏占位
- 管理端：管理员登录、RBAC、用户/服务方/订单/内容/投诉/配置接口分组

## 4. 核心业务闭环
1. 用户登录后浏览服务或内容
2. 用户创建订单并发起支付
3. 服务方接单并执行履约动作
4. 用户确认完成，进入评价/投诉/结算后链路
5. 管理端进行审核、干预、风控与权限治理

## 5. 下一阶段优先级
1. 完善 Web 与 App 的真实页面，不再停留在骨架页
2. 将 AI、音频等仍偏占位模块推进为真实业务实现
3. 补齐审计日志、风险事件与后台治理闭环
4. 支付、退款、结算、提现等资金链路继续深化，但开发优先级放在最后
