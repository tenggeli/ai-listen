# listen 数据库设计

## 1. 文档目标
- 明确 listen 项目在 MVP 到第二阶段的核心数据库模型。
- 统一表结构命名、字段规则、索引策略和状态设计。
- 为后续 MySQL 建表脚本、ORM 模型、接口开发提供依据。

## 2. 设计原则
- 数据库选型：MySQL 8.0+
- 字符集：`utf8mb4`
- 排序规则：`utf8mb4_general_ci`
- 时间字段统一使用 `datetime`
- 主键统一使用 `bigint unsigned`
- 金额统一使用“分”为单位的整数类型
- 关键业务保留状态流转日志，避免仅靠当前状态字段追踪
- 所有核心表建议保留：
  - `id`
  - `created_at`
  - `updated_at`
  - `deleted_at`

## 3. 命名规范

### 3.1 表命名
- 使用小写下划线复数形式，如 `users`、`orders`
- 中间表命名采用“主体_关系_客体”风格，如 `provider_service_items`

### 3.2 字段命名
- 主键统一：`id`
- 外键统一：`xxx_id`
- 状态字段统一：`status`
- 布尔标记统一：`is_xxx`
- 金额字段统一：`amount_xxx` 或明确业务语义，如 `pay_amount`

## 16. 分库分表与扩展建议
- MVP 阶段保持单库。
- 当订单表、日志表、播放日志表增长较快时，优先按时间或业务维度归档。
- 高频写入表如 `order_status_logs`、`audio_play_logs` 可先做冷热分层。

## 17. 建表优先顺序
1. 用户相关表
2. 服务提供方相关表
3. 服务项目表
4. 订单、支付、结算、评价、投诉表
5. 后台审核和系统配置表
6. 广场与声音内容表

## 18. 结论
- 这份数据库设计优先服务 MVP 闭环，同时为广场、声音、VIP、推荐和风控预留扩展空间。
