# listen 接口设计规范

## 1. 文档目标
- 统一 listen 项目的前后端接口风格、路径规划、鉴权方式、错误码和模块划分。
- 输出 MVP 阶段核心接口清单，便于后续后端实现、前端联调和测试编写。

## 2. 总体约定

### 2.1 接口风格
- 以 RESTful API 为主
- 实时聊天、订单状态推送可扩展 WebSocket/SSE
- 统一前缀：`/api/v1`
- 管理后台统一前缀：`/api/v1/admin`

### 2.2 请求头约定
- `Content-Type: application/json`
- `Authorization: Bearer <token>`
- `X-Request-Id: <uuid>` 可选，便于链路追踪
- `X-Device-Id: <string>` 可选，便于风控识别

### 2.3 响应结构

```json
{
  "code": 0,
  "message": "ok",
  "data": {},
  "traceId": "9f2c3a1b8d7141c5"
}
```

说明：
- `code = 0` 表示成功
- 非 `0` 表示业务失败
- `traceId` 用于日志追踪

### 2.4 分页结构

```json
{
  "list": [],
  "pagination": {
    "page": 1,
    "pageSize": 20,
    "total": 100
  }
}
```

## 3. 鉴权与权限

### 3.1 用户端
- 登录成功后返回 `accessToken` 和 `refreshToken`
- 用户端接口通过 JWT 鉴权

### 3.2 服务提供方
- 服务提供方本质仍是用户账号
- 判断是否具备服务方身份和审核状态
- 仅服务方可访问工作台、接单、提现等接口

### 3.3 后台管理员
- 后台接口使用独立管理员登录体系
- 采用 RBAC 权限模型

## 4. 错误码规范

### 4.1 通用错误码
| 错误码 | 说明 |
|---|---|
| 0 | 成功 |
| 40000 | 请求参数错误 |
| 40100 | 未登录或 token 无效 |
| 40300 | 无权限访问 |
| 40400 | 资源不存在 |
| 40900 | 状态冲突 |
| 42900 | 请求过于频繁 |
| 50000 | 系统异常 |

### 4.2 业务错误码
| 错误码 | 说明 |
|---|---|
| 100101 | 验证码错误或已过期 |
| 100102 | 用户已存在 |
| 100201 | 服务方审核未通过 |
| 100202 | 服务方当前不可接单 |
| 100301 | 订单状态不允许当前操作 |
| 100302 | 订单不存在 |
| 100303 | 支付状态异常 |
| 100304 | 余额不足 |
| 100401 | 提现审核中 |
| 100501 | 内容违规 |
| 100601 | AI 推荐次数已达上限 |

## 5. 幂等与安全要求
- 创建订单、支付、退款、提现等写接口必须支持幂等
- 建议请求头传 `Idempotency-Key`
- 涉及金额变化的接口必须记录操作日志
- 支付回调接口必须进行签名校验

## 9. 接口实现优先级

### P0
- 认证模块
- 用户模块
- 服务方入驻和工作台基础接口
- 服务项目接口
- 订单主链路接口
- 支付接口
- 后台审核与订单管理接口

### P1
- 提现、评价、投诉
- 广场动态
- 声音模块

### P2
- VIP、礼物、消息中心
- WebSocket 实时能力
- AI 个性化增强接口

## 10. 前后端联调建议
- 先产出 OpenAPI 文档或 Apifox 集合
- 接口返回字段命名统一使用小驼峰
- 时间字段统一返回 `YYYY-MM-DD HH:mm:ss`
- 金额字段优先返回“分”和格式化后的展示字段
