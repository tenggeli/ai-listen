# listen 开发规范与面向对象设计（精简版）

## 1. 目标
- 控制复杂度：职责清晰、边界稳定、可渐进重构
- 保持四端一致：同业务语义、同状态定义、同错误码约定
- 支撑双存储：内存与 MySQL 行为一致

## 2. 后端规范

### 2.1 分层约束
- `routes`：仅做路由分组和中间件
- `controller`：参数校验、鉴权、协议转换
- `service`（新增功能优先引入）：业务编排与状态流转
- `model/store`：数据访问与持久化

### 2.2 依赖方向
- 允许：`routes -> controller -> service -> model`
- 禁止：`model -> controller`、`dto -> model`、跨模块直接改内部状态

### 2.3 代码组织
- 轻模块：`routes.go + controller.go (+ dto.go)`
- 复杂模块：`routes.go + *_controller.go (+ dto.go + service.go)`
- 状态型模块（订单、支付、审核、投诉）必须显式维护状态枚举和流转规则

## 3. 前端规范（Web + App）
- 页面只做展示和交互，不直接堆业务编排
- SDK 层统一调用接口，避免页面内散落请求逻辑
- 可复用能力优先放到共享层（`app/packages/shared`）
- 用户端与管理端都要复用同一后端语义，不自创状态字段

## 4. 多端一致性规则
- “可裁剪，不分叉”：页面可简化，接口语义不可分裂
- 同一动作命名保持一致（如接单、出发、到达、完成）
- 管理 App 与管理 Web 必须复用同一 RBAC 权限点

## 5. 测试与文档
- 最低要求：新增状态流转必须补关键测试
- 控制器测试优先覆盖：鉴权、参数错误、权限错误
- 支付回调类接口必须覆盖验签、时效、幂等和重复通知场景
- 新增模块必须同步更新 `doc/` 对应文档，不留“文档债务”
