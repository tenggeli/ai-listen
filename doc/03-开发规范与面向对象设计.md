# listen 开发规范与面向对象设计

## 1. 目标
- 统一团队开发方式，降低多人协作时的理解成本和维护成本
- 以面向对象思想组织业务能力，即使在 Go 语言中不强调传统类，也要强调对象职责、边界和协作关系
- 保证新增模块具备可扩展、可测试、可替换、可复用的特性

## 2. 基本原则

### 2.1 面向对象优先
- 先识别业务对象，再设计接口、流程和数据结构
- 每个对象只承担单一核心职责，避免一个对象同时处理鉴权、编排、存储和返回组装
- 对象之间通过接口协作，不直接依赖具体实现
- 优先组合而非继承，使用结构体组合、接口抽象和依赖注入实现能力复用

### 2.2 高内聚低耦合
- 模块内聚焦一个业务域，例如 `order`、`auth`、`provider-center`
- 跨模块调用只能依赖公开能力，不允许直接操作对方内部状态
- Handler 不直接承载复杂业务规则，复杂逻辑必须下沉到 Service 或 Domain 对象

### 2.3 依赖倒置
- 业务层依赖抽象，不依赖 MySQL、内存存储、第三方 SDK 等具体细节
- 存储、支付、消息、AI 能力都应先定义接口，再提供实现
- 允许通过替换实现完成本地调试、测试和生产切换

## 3. 分层规范

### 3.1 推荐分层
- `Handler`：负责路由、参数解析、鉴权结果获取、响应输出
- `Service`：负责业务编排、规则校验、事务边界、状态流转
- `Repository` / `Store`：负责数据访问与持久化
- `Domain Model`：负责表达核心业务对象、状态和值对象
- `DTO`：负责输入输出传输，不承载业务行为

### 3.2 分层职责要求
- Handler 只做协议转换，不写核心业务规则
- Service 只编排当前业务，不关心 HTTP 细节
- Store 只处理数据读写，不掺杂业务状态判断
- DTO 不能直接当数据库模型使用，也不能替代领域对象

### 3.3 依赖方向
- `handler -> service -> store`
- `handler -> dto`
- `service -> domain / repository interface`
- 禁止 `store -> handler`
- 禁止 `dto -> store`
- 禁止跨模块直接访问别的模块私有实现

## 4. 对象设计规范

### 4.1 先抽象业务对象
- 每个业务模块开发前，先明确核心对象、职责、状态和行为
- 例如订单域至少应识别：`Order`、`Payment`、`OrderStatusLog`、`OrderService`
- 例如后台权限域至少应识别：`AdminUser`、`Role`、`Permission`、`AuthorizationService`

### 4.2 对象职责清晰
- 实体对象负责维护自身关键状态和约束
- 值对象负责表达不可变业务概念，如手机号、金额、地理位置、时间区间
- 领域服务负责封装不适合放在单个实体中的规则
- 工厂对象负责复杂对象创建，避免在 Handler 中拼装大量字段

### 4.3 面向接口编程
- 对外暴露能力时优先定义接口，如 `OrderRepository`、`PaymentGateway`、`TokenIssuer`
- 接口命名以业务语义为主，不以技术细节为主
- 一个接口只暴露当前调用方真正需要的方法，避免“大而全”接口

## 5. 编码规范

### 5.1 命名规范
- 包名使用简洁小写单词，优先使用业务名词
- 结构体命名使用业务语义，如 `OrderService`、`ProviderApplication`
- 接口命名优先使用业务角色或能力名，避免滥用 `Manager`、`Util`
- 方法命名必须体现动作和意图，如 `CreateOrder`、`ApproveProvider`

### 5.2 文件组织规范
- 一个模块内建议按职责拆分文件：`handler.go`、`service.go`、`repository.go`、`dto.go`、`model.go`
- 当模块变复杂时，可以继续拆成子文件，但必须保证目录语义稳定
- 公共能力放在 `pkg` 或 `internal/support`，禁止把业务逻辑沉淀成无语义工具函数

### 5.3 方法长度与复杂度
- 单个方法应保持短小，超过一个屏幕高度时优先拆分
- 复杂分支逻辑应拆为私有方法或策略对象
- 禁止在一个方法中同时处理参数校验、业务规则、数据库写入和响应拼装

## 6. 模块开发约束

### 6.1 新增业务模块流程
1. 先补业务对象设计和职责说明
2. 再定义 DTO、Service 接口、Repository 接口
3. 最后接入 Handler、Store 实现和路由

### 6.2 状态型业务要求
- 订单、支付、审核、投诉、退款等状态型业务必须显式定义状态机
- 状态流转规则统一下沉到 Service 或 Domain 对象
- 不允许在多个 Handler 或多个 Store 实现里重复维护状态判断

### 6.3 扩展点设计
- 支付、短信、AI 推荐、文件存储等外部能力都必须预留接口层
- 新接三方平台时，原则上只新增实现，不改调用方主流程

## 7. 测试规范
- Service 层必须优先覆盖单元测试
- 涉及状态流转的对象必须覆盖正常流转、非法流转和边界场景
- Store 层测试关注持久化一致性，Handler 层测试关注协议与鉴权
- 新增接口时，至少补一条成功路径和一条失败路径测试

## 8. 当前项目落地要求

### 8.1 后端 Go 项目
- 当前 `internal/modules/*` 建议逐步从“仅 Handler”模式演进为“Handler + Service + Store Interface”模式
- `store.Default()` 适合作为过渡方案，后续应逐步改为显式依赖注入，降低全局状态耦合
- 订单、支付、后台 RBAC 等核心域优先补齐 Service 对象和领域规则封装

### 8.2 前端项目
- 页面开发也应遵循对象化思维，按“页面容器、业务模型、接口服务、展示组件”分层
- 禁止把接口请求、状态编排和页面展示全部堆在单个 Vue 文件中
- 复杂表单、订单状态展示、后台权限管理应抽离独立 service / model 层

## 9. 禁止事项
- 禁止面向过程式堆砌代码，出现超长 Handler 或超长页面文件
- 禁止为了赶进度直接跨层调用数据库或直接拼 SQL 到业务逻辑里
- 禁止把通用 `map[string]any` 当作长期业务对象使用
- 禁止复制粘贴相似逻辑而不抽象公共对象和接口

## 10. 文档执行要求
- 后续新增模块时，必须先补最小设计说明，再开始编码
- PR 或提测前需自检是否满足单一职责、分层清晰、依赖抽象三个基本要求
- 若现有历史代码不满足本规范，新需求开发时应同步做小步重构，不继续放大结构债务
