# listen 技术方案设计

## 1. 技术目标
- 支撑 App、Web、自适应 H5、后台管理多端统一开发。
- 以 Go + MySQL + Vue + Nginx 为主技术栈，兼顾扩展性、稳定性和后续 AI 能力接入。
- 面向订单、履约、支付、审核和风控等核心业务，优先保证可靠性与可审计性。

## 2. 总体技术架构

### 2.1 架构原则
- 先单体分层，后按业务演进为模块化服务。
- 前后端分离，统一 API 网关入口。
- 核心业务状态机化，避免订单状态混乱。
- 配置中心化、日志结构化、监控指标化。

### 2.2 推荐架构分层
- 接入层：Nginx、HTTPS、静态资源分发、反向代理。
- 前端层：Vue 3 用户端 Web/H5、Vue 3 管理后台、App 外壳。
- 应用层：Go API 服务、管理服务、定时任务、消息通知服务。
- 业务层：用户、服务提供方、订单、支付、内容、推荐、风控、报表模块。
- 数据层：MySQL、Redis、对象存储、日志存储。
- 集成层：短信、微信登录、支付、地图定位、实名认证、AI 模型服务。

## 3. 终端形态与实现建议

### 3.1 App 实现建议
- 方案建议：`uni-app x` 或 `uni-app + Vue3` / `Flutter` 二选一。
- 若希望严格围绕 `go + mysql + vue`，建议优先 `uni-app + Vue3`。
- 原因：
  - 与 Web/管理后台技术栈统一。
  - 开发效率高，适合 MVP 快速上线。
  - 可同时覆盖 iOS、Android、H5。

### 3.2 Web 与管理后台
- 用户端自适应 Web：Vue 3 + Vite + TypeScript。
- 管理后台：Vue 3 + Element Plus 或 Naive UI。
- 状态管理：Pinia。
- 路由：Vue Router。
- 网络层：Axios 或基于 `fetch` 的统一封装。

## 4. 后端技术选型

### 4.1 核心框架
- Go 版本：Go 1.23+
- Web 框架：Gin
- ORM：GORM 或 sqlc + 原生 SQL
- 配置管理：Viper
- 日志：Zap
- 参数校验：validator
- 定时任务：robfig/cron

### 4.2 推荐模块划分
- `api-gateway`：统一接入、鉴权、限流、日志透传
- `user-service`：用户资料、认证、标签、收藏、VIP
- `provider-service`：服务提供方入驻、资质、状态、项目与报价
- `order-service`：订单创建、状态流转、计时、取消、评价
- `payment-service`：支付单、退款、结算、提现
- `content-service`：广场动态、声音内容、评论点赞
- `admin-service`：后台管理接口
- `ai-service`：AI 对话、推荐编排、智能追问、标签抽取
- `job-service`：异步任务、超时处理、结算任务、通知补偿

### 4.3 单体优先的目录思路
- 第一阶段建议采用模块化单体：
  - `cmd/server`
  - `internal/modules/user`
  - `internal/modules/provider`
  - `internal/modules/order`
  - `internal/modules/payment`
  - `internal/modules/content`
  - `internal/modules/admin`
  - `internal/modules/ai`
  - `pkg/`
- 这样更适合 MVP 阶段控制复杂度。

  - 用户表

  - 提现单表

  - 广场动态表
  - 声音内容表

- 验证码

### 5.3 对象存储

## 6. 核心业务模型设计

### 6.1 用户模型
- 基础身份信息
- 性别、年龄、兴趣标签
- 城市定位
- VIP 状态
- 钱包信息
- 行为画像标签

### 6.2 服务提供方模型
- 实名与审核状态
- 服务项目和价格
- 当前接单状态
- 评分、积分、服务次数、取消率
- 收款账户
- 标签与擅长场景

### 6.3 订单状态机
- 待支付
- 待接单
- 已接单
- 已出发
- 已到达
- 服务中
- 待确认完成
- 已完成
- 已取消
- 申诉中
- 已退款

### 6.4 结算模型
- 平台托管资金
- 完单后进入待结算
- 正常订单 T+1 结算
- 异常订单挂起结算
- 提现审核与打款回执留痕

## 7. AI 能力方案

### 7.1 AI 使用场景
- 首页语音/文字需求理解
- 场景标签抽取
- 智能推荐 3 位候选服务提供方
- 智能追问与偏好学习
- 文案辅助生成
- 广场/评价/聊天内容风控预判

### 7.2 AI 架构建议
- 采用独立 `ai-service` 作为编排层，不将模型调用散落在各业务模块。
- 核心流程：
  - 输入解析
  - 意图识别
  - 标签抽取
  - 规则过滤
  - 候选集召回
  - 排序推荐
  - 推荐理由生成

### 7.3 推荐策略建议
- 规则优先，模型增强。
- 首期排序维度：
  - 城市与距离
  - 服务类目匹配度
  - 当前状态
  - 价格区间
  - 评分与投诉率
  - 历史偏好
- 后续增加：
  - 协同过滤
  - 人群画像相似度
  - 订单完成率反馈学习

## 8. API 设计原则
- RESTful 为主，必要时对聊天和状态推送使用 WebSocket/SSE。
- 所有写接口要求幂等设计。
- 返回体统一结构：`code`、`message`、`data`、`traceId`。
- 后台接口与客户端接口隔离。
- 敏感接口必须记录审计日志。

## 9. 安全与权限设计

### 9.1 鉴权
- 用户端：JWT + Refresh Token。
- 管理后台：JWT + RBAC 权限模型。
- 关键操作二次校验：提现审核、实名审核、退款仲裁。

### 9.2 数据安全
- 手机号、身份证号、银行卡号脱敏展示。
- 密码、密钥、支付配置采用环境变量和密钥管理。
- 文件上传进行类型校验和防病毒扩展预留。

### 9.3 风控能力
- 登录、验证码、支付、发布内容限流。
- 黑名单、设备指纹、异常行为监控。
- 投诉工单、取消原因、人工复核闭环。

## 10. 音视频与实时能力
- 语音输入：优先调用移动端原生能力或第三方语音识别服务。
- 实时消息：聊天和订单状态建议使用 WebSocket。
- 推送通知：App 使用极光/个推/Firebase APNs 等。
- 声音播放：音频文件存储在对象存储，经 CDN 分发。

## 11. 部署方案

### 11.1 环境规划
- 开发环境 dev
- 测试环境 test
- 预发环境 staging
- 生产环境 prod

### 11.2 部署结构
- Nginx：静态资源、HTTPS、反向代理
- Go 服务：systemd 或 Docker 部署
- MySQL：主从或云数据库
- Redis：独立实例
- 对象存储：云 OSS/COS

### 11.3 推荐部署方式
- 第一阶段：Docker Compose
- 第二阶段：Kubernetes 或云容器平台

## 12. 运维与可观测性
- 日志：结构化日志，按服务和 traceId 查询。
- 指标：QPS、接口耗时、订单转化率、支付成功率、推荐命中率。
- 告警：支付异常、接口错误率升高、队列积压、数据库连接异常。
- 链路追踪：OpenTelemetry。

## 13. 测试策略
- 单元测试：核心领域逻辑、状态机、金额计算。
- 集成测试：下单、支付、完单、退款、提现链路。
- 接口测试：OpenAPI + 自动化测试集合。
- 前端测试：组件测试、关键页面 E2E。
- 压测重点：推荐接口、订单创建、支付回调、内容流接口。

## 15. 开发阶段建议

### 15.1 第一阶段
- 搭建前后端基础工程
- 完成登录、用户、服务提供方、项目、订单、后台基础能力

### 15.2 第二阶段
- 完成支付、结算、提现、评价、投诉
- 接入 AI 推荐基础版本

### 15.3 第三阶段
- 完成广场、声音、消息推送、VIP、打赏
- 增强风控和运营体系

## 16. 技术风险与应对
- 风险：订单状态复杂导致逻辑混乱
  - 应对：状态机统一建模，禁止前端拼装业务状态
- 风险：支付与结算差错
  - 应对：幂等、回调验签、对账任务、人工补偿流程
- 风险：AI 推荐不稳定
  - 应对：规则兜底、推荐缓存、人工运营位
- 风险：内容合规压力
  - 应对：机审 + 人审 + 举报闭环

## 17. 技术结论
- listen 适合采用“Vue 多端 + Go 模块化单体 + MySQL/Redis + Nginx”的方案快速起盘。
- 在 MVP 阶段优先保证订单、审核、支付、结算和风控闭环，AI 与声音内容采用可渐进增强的模块设计。
