# listen 技术方案设计

## 1. 技术目标
- 支撑 App、Web、自适应 H5、后台管理多端统一开发。
- 以 Go + MySQL + Vue + Nginx 为主技术栈，优先保证订单、审核、支付等核心业务的稳定性。
- 在 MVP 阶段采用模块化单体，降低系统复杂度，保留后续拆分空间。

## 2. 总体架构

### 2.1 架构原则
- 先单体分层，后按业务演进为模块化服务。
- 前后端分离，统一 `/api/v1` 作为接口入口。
- 核心交易链路状态机化，避免订单状态混乱。
- 配置通过环境变量注入，日志结构化输出。

### 2.2 当前代码结构
- 接入层：Gin 路由与 HTTP Handler
- 应用层：`internal/app` 负责配置、日志、存储初始化
- 业务层：`internal/modules/*` 按认证、用户、服务方、订单、支付、后台等模块组织
- 数据层：`internal/store` 提供统一存储抽象，支持内存实现和 MySQL 实现
- 部署层：`deploy/nginx`、`deploy/docker`

## 3. 前端方案

### 3.1 当前实现
- `web/apps/user-web`：Vue 3 + Vite 用户端骨架
- `web/apps/admin-web`：Vue 3 + Vite 管理后台骨架
- `app/listen-app`：App 工程占位

### 3.2 建议方向
- 若继续坚持 `Go + MySQL + Vue` 技术统一，App 建议优先采用 `uni-app + Vue3`
- 管理后台后续可接入成熟组件库，如 Element Plus 或 Naive UI

## 4. 后端方案

### 4.1 技术选型
- Go 1.23+
- Gin
- Viper
- Zap
- `database/sql + 原生 SQL`

### 4.2 当前模块
- `auth`：短信验证码、短信登录、刷新 token
- `user`：当前用户信息与基础资料修改
- `provider`：服务方列表与详情
- `provider-center`：入驻申请、资料配置、服务项配置、履约动作
- `order`：创建、取消、开始、确认完成、状态日志
- `payment`：创建支付单
- `admin`：服务方审核与后台占位接口
- `aimatch`、`review`、`complaint`、`post`、`audio`、`me`：已注册路由，当前多为占位实现

## 5. 存储方案

### 5.1 MySQL
- 用于存储核心业务数据
- 当前已落地的核心表包括：
  - `users`
  - `user_auths`
  - `auth_sms_codes`
  - `auth_tokens`
  - `providers`
  - `provider_profiles`
  - `provider_service_items`
  - `service_items`
  - `orders`
  - `order_status_logs`
  - `payments`
  - `audit_records`

当前实现说明：
- 配置 `MYSQL_DSN` 后启用 MySQL 存储
- 应用启动时会自动执行初始化建表 SQL
- 启动时会补齐基础服务项目种子数据

### 5.2 内存存储
- 未配置 `MYSQL_DSN` 时默认启用
- 用于本地快速调试和测试
- 当前路由级测试默认使用内存存储

### 5.3 Redis 与对象存储
- 设计上仍建议后续引入 Redis 处理验证码、限流、幂等键与缓存
- 图片、音频、实名附件等文件资源后续建议接入对象存储
- 当前代码尚未接入这两类外部依赖

## 6. 核心业务模型

### 6.1 认证模型
- 短信验证码写入 `auth_sms_codes`
- 登录成功后写入 `user_auths`
- access token 与 refresh token 写入 `auth_tokens`
- 当前鉴权方式为 Bearer Token + 服务端持久化 token 校验

### 6.2 服务提供方模型
- `providers` 维护服务方主状态
- `provider_profiles` 维护展示资料
- `provider_service_items` 维护服务项目与报价
- `audit_records` 记录审核操作留痕

### 6.3 订单状态机
- 待支付
- 待接单
- 已接单
- 已出发
- 已到达
- 服务中
- 待确认完成
- 已完成
- 已取消

当前代码已实现以下流转：
- 创建订单 -> 待支付
- 支付成功 -> 待接单
- 服务方接单 -> 已接单
- 服务方出发 -> 已出发
- 服务方到达 -> 已到达
- 用户确认开始 -> 服务中
- 服务方完单 -> 待确认完成
- 用户确认完成 -> 已完成
- 用户取消 -> 已取消

## 7. API 设计原则
- RESTful 为主
- 统一返回 `code`、`message`、`data`、`traceId`
- 用户端与后台端接口隔离
- 订单状态流转只允许通过后端控制

当前未完成项：

- 请求级 trace 透传
